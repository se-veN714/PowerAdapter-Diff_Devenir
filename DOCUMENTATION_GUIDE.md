# DOCUMENTATION_GUIDE.md — PAdif 文档治理

> **文档权限**: 本文是 PAdif 全部 Markdown 文档的**权威源与唯一裁决处**——定义阅读顺序、各文档职责边界与冲突解决规则。其余文档不在自身重复声明「本文档为权威」式冗余注释，统一以本文为准。

本文档本身不随功能迭代频繁改动，仅在文档结构 / 职责调整时更新。

---

## 1. 文档清单与职责（权威域）

| 文档 | 职责（权威域） | 阅读时机 |
|------|----------------|----------|
| `CONTEXT.md` | 项目状态快照：当下阶段 / 决策日志 / 下一步。本地**非 git 跟踪**，仅作快速入场 | 回到项目、想知道「现在到哪了 / 为什么这么定 / 下一步干啥」时 |
| `GUIDE-PAdif.md` | 产品需求与设计意图的权威源：需求上下文、提交模型、diff 模型、形态路线、任务权重 | 理解「为什么要做、做成什么样」时 |
| `DEVELOPMENT.md` | 架构 / 编码约束 / API / 数据模型的权威源（**技术事实唯一裁决**） | 动手改代码、确认接口 / 约束 / 数据结构时 |

> `LICENSE`、`samples/*.md` 为许可与测试 fixture，不在此治理范围内。

---

## 2. 阅读顺序

`CONTEXT.md`（30 秒入场）→ `GUIDE-PAdif.md`（理解产品意图）→ `DEVELOPMENT.md`（落地约束与接口）。

---

## 3. 冲突解决

- **技术事实**（接口签名、配置项、数据结构、已知行为、Mermaid 图）以 **`DEVELOPMENT.md`** 为准。
- **产品意图 / 需求取舍**以 **`GUIDE-PAdif.md`** 为准。
- **状态 / 进度**以 **`CONTEXT.md`** 为准（但其为本地快照，架构 / 约束的权威仍以 DEVELOPMENT 为准）。
- 三处不一致时，先回到本文确认职责边界，再修正对应文档；**禁止**在各文档内重复声明冗余权威注释。

---

## 4. 文档维护规则（源自 dev-doc-writer 技能）

1. **如实反映已提交代码**：文档必须匹配当前**已提交**实现（含迁移 / diff）。规划中、未实现的功能一律用 `TODO` 明确标记，不得写成「已完成」。
2. **版本记录**：每次架构 / 接口变更须同步更新 `DEVELOPMENT.md` 的版本记录（Section 0）与对应 Mermaid 图。
3. **图一律用 Mermaid**，禁止 ASCII art；Mermaid 语法须可渲染（节点标签避免 `<` `>` 等需转义字符）。
4. **能力项归属**：每项能力须在对应文档的权限域内描述，权限注释保持唯一、不重复。
5. **未实现项用 TODO**：`- [ ] **范围 / 项**: 说明。` 格式，并归属到对应文档权限域内，成为一体。
