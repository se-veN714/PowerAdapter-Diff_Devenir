# GUIDE-PAdif —— 轻量化文章版本监管与差异工具

> PAdif = **P**aragraph / **A**rticle **Dif**f
> 一个 git 式、但为「散文/长文」而非「代码」设计的版本控制小工具。
> 当前阶段：先写网页应用，后续可迭代为 Obsidian 插件。

---

## 0. 需求上下文摘要（来自用户的回答）

在动手前，用户（深度内容创作者，使用 Obsidian 管理「世界是肺」世界观）明确了四个关键约束：

| 维度 | 用户的选择 | 关键原话 / 含义 |
|------|-----------|----------------|
| **内容来源** | Markdown 文件 | 直接管理 `.md` 文件，按文件记录版本（与 Obsidian 笔记库天然契合）。 |
| **版本触发** | 类似 git 的「提交 + commit message」，并采用 **major.minor.patch 语义版本标注** | 用户补充：「不用限制太多……类似现实版本管理那样 version-patch 的形式（major.minor.patch），辅助创作者适当标注。不要太过严格，相信创作者有自己的小巧思，仅作引导。」→ 用语义版本**引导**创作者区分改动规模，而非强制拦截。 |
| **差异算法** | 句子/标点分段，而非按行 | 用户原话：「文章有个特点就是不像代码那样分行的，所以我们的差异计算应该是以句号、分号或者其他符号作为分段标准。」→ 参考 Python `difflib`，但分段单位改为语义标点。 |
| **应用形态** | 先网页，后 Obsidian 插件 | 用户原话：「先写网页，后续可迭代到 Obsidian 插件。」→ 第一阶段交付本地网页应用（浏览器打开、零安装）。 |

## 1. 产品定位

- **它是什么**：一个轻量的「文章版本库」。你像 git commit 一样，在每次有意义的改动后「提交一版」，并写一句改动说明；之后随时可以挑任意两个版本，看到它们之间**精确到句子级别**的差异。
- **它不是什么**：不是通用 Git 客户端，不处理分支/合并/冲突；不为代码设计；不追求重量级协作。
- **核心痛点**：散文修改是连续的、不分行的，普通文本 diff（按行）会把整段标红、看不出到底改了哪几个字。PAdif 以「句/标点」为最小单位，让「这一版到底改了什么」一目了然。

## 2. 核心需求清单

### 2.1 内容模型
- 管理的对象是 **Markdown 文件**（`.md`）。
- 每个被纳入监管的 `.md` 文件 = 一个 **Article（文章）**。
- 每个 Article 有一串 **Version（版本）**，按时间顺序。

### 2.2 提交（Commit）模型
- 每次提交需要：
  1. 当前文件内容（快照）；
  2. 一条 **commit message**（必填、非空即可，不过度限制长度/格式 —— 信任创作者的表达）；
  3. 一个 **语义版本标注 major.minor.patch**，用于辅助区分本次改动规模；
  4. 时间戳、来源标记。
- **语义版本标注引导（重要，但仅引导不强制）**：
  - 提供 major / minor / patch 的语义提示，帮助创作者判断该打哪一级：
    - **major（主版本）**：结构/论点/主题的大改，文章「换了一个样子」。
    - **minor（次版本）**：新增或删减了成段内容、章节、重要论据。
    - **patch（修订）**：措辞润色、标点修正、错字修订等小幅调整。
  - 系统**仅作引导，不做强校验、不拒绝提交**：相信大多数深度创作者有自己的小巧思，即使改动「看起来小」也可能很有意义。
  - 版本号可由系统根据创作者选择的级别**自动递增**（如上一版 `1.2.3`，选 patch → `1.2.4`），也允许手动填写。
  - （可选）若检测到本次相对上一版内容几乎无变化，给出一个**温和提醒**（如「这版似乎只是标点/空格变化，确定要提交吗？」），但**不阻断**提交。

### 2.3 差异计算（Diff）模型
- **分段单位**：以中文/英文的句末与句中标点作为切分边界，例如：
  - 句末：`。 ！ ？ . ! ?` 以及段落换行 `\n\n`
  - 句中可作为次级切分：`， ； 、 ； , ; :`
- **算法**：基于 `difflib`（Python）或等价算法（Node 端可用 `diff` / `jest-diff` 思路），但**输入不是「行」，而是「句子/片段」数组**。
- **输出粒度**：
  - 默认 **逐句/逐词高亮**：新增的句子标绿、删除的标红、改动的句子内部做词级对齐高亮。
  - （后续）并排双栏、统计摘要（字数增减、段落增减、句子增减）。

### 2.4 形态与交付
- 第一阶段：**本地网页应用**。起一个本地服务，浏览器打开即用的单页应用。
- 第二阶段（规划中）：封装为 **Obsidian 插件**，直接嵌在笔记软件里，复用同一套版本库数据。

## 3. 关键设计决策（已锁定）

1. **Diff 以「句子」而非「行」为最小单位** —— 由散文无分行的特性决定。分段正则优先按 `。！？.!?` 与空行断句，逗号/分号作为次级对齐。
2. **提交带 message + 语义版本（major.minor.patch）标注，但仅做引导、不做强校验** —— 由用户「不要限制太多、相信创作者小巧思、仅作引导」的诉求决定。版本号作温和提示与自动递增，不阻断提交。
3. **先网页后插件** —— 网页的形态复用同一份本地版本库（如 SQLite 或 JSON 文件），将来 Obsidian 插件只需换 UI 层、复用存储与 diff 引擎。
4. **轻量优先** —— 不引入重型依赖与复杂架构；本地存储、单机使用、零云依赖。

## 4. 架构草案（第一阶段：网页应用）

```
padif/
├── GUIDE-PAdif.md          # 本文档
├── server/                 # 本地后端（读取 .md、存版本、做 diff）
│   ├── app.py              # 轻量 Web 服务（推荐 FastAPI / Flask）
│   ├── store.py            # 版本存储（SQLite 或 JSON）
│   ├── differ.py           # 句子级 diff 引擎（基于 difflib + 自定义分段）
│   └── version.py          # 语义版本（major.minor.patch）标注与自动递增引导
├── web/                    # 前端单页（HTML/CSS/JS，轻框架或原生）
│   ├── index.html
│   ├── app.js              # 文章列表 / 提交 / 版本对比 UI
│   └── diffview.js         # 高亮渲染
└── data/                   # 版本库数据（.md 原文 + 版本快照 + 元数据）
```

### 4.1 数据模型（初版）
- `Article`: `{ id, path, title, created_at, current_version_id }`
- `Version`: `{ id, article_id, content, commit_message, version, version_kind, created_at, diff_stats }`
  - `version`：语义版本号字符串，如 `1.2.3`
  - `version_kind`：本次标注级别，`major` / `minor` / `patch`
- `diff_stats`: `{ chars_added, chars_removed, sentences_added, sentences_removed }`

### 4.2 技术选型（已确认）

- **后端**：Python 3.12（已具备 managed runtime）。`FastAPI` + `difflib`。
- **前端 ↔ 后端耦合**：**htmx**（用户确认，替代 Alpine.js）。HTML 片段由后端模板渲染，htmx 负责无刷新交互，避免重构建链、保持轻量。
- **存储**：**SQLite（已确认）**。

  > **关于 MongoDB 的考量（用户提出，结论：MVP 阶段用 SQLite）**
  > - MongoDB 是 NoSQL 文档库，其优势在水平扩展、分布式、海量灵活 schema——对「单机、单用户、本地、轻量」的 PAdif 属于过度配置。
  > - 本工具的数据是**强结构化关系型**：Article 1—∞ Version，版本带快照/版本号/差异统计。这正是 SQLite 的舒适区；且 SQLite 是**单文件、零服务进程、零安装**，最契合「轻量化 / 网页 MVP 先跑起来」的目标。
  > - 若未来确实需要文档形态（如把整篇文章+版本历史作为一个大文档），SQLite 同样可用 JSON 字段承载，无需引入服务器。
  > - 关键约束：**MVP 先用最小技术栈跑通，再按需增加技术栈强化功能**（用户原则）。MongoDB 会引入 `mongod` 常驻进程，违背「零安装」，故 MVP 不采用；后续若形态演化再评估。
- **开发原则（用户确认）**：先以网页实现 MVP（最小可用），之后**逐步增加技术栈**来强化与增幅功能，而非一开始就堆砌重型依赖。
- **实现偏差说明（已落地）**：原选型后端为 FastAPI，但当前 managed Python 环境无法安装第三方包（OpenSSL/网络限制），MVP 已改用 **标准库 `http.server`** 实现（零依赖、保证可运行），路由与业务逻辑与框架解耦，后续可平滑迁回 FastAPI。详见 `DEVELOPMENT.md` 头部与「环境限制」注。

## 7. 任务权重与分阶段实现

按「先核心后增强」排布，权重代表相对工作量/风险。实现顺序遵循权重与依赖关系。

> **状态（2026-07-07）**：P0（文档与规范）已完成；P1（MVP）已实现并自测通过——导入 .md → 提交（带语义版本）→ 选两版看句子级 diff 全链路跑通。后续为 Phase 2–4 增幅。

| 编号 | 任务 | 模块 | 权重 | 依赖 | 阶段 |
|------|------|------|------|------|------|
| T1 | 句子级 diff 引擎（按标点分段 + 逐句/逐词高亮） | `server/differ.py` | **30%**（最高，核心难点） | — | MVP |
| T2 | SQLite 存储层（Article/Version schema + 读写） | `server/store.py` | 15% | — | MVP |
| T3 | 语义版本标注引导（major.minor.patch + 自动递增 + 温和提醒） | `server/version.py` | 10% | T2 | MVP |
| T4 | 后端 API（导入 md / 提交 / 列版本 / 版本对比） | `server/app.py` | 20% | T1, T2, T3 | MVP |
| T5 | 前端 htmx UI（列表 / 提交 / diff 高亮渲染） | `web/*` | 20% | T4 | MVP |
| T6 | 开发者文档与编码规范 | `DEVELOPMENT.md` | 5% | — | 贯穿 |
| T7 | 前端优化（样式打磨、交互体验、响应式 / 无障碍、性能收尾等） | `web/*` | **3%**（最低，留待收尾统一处理） | T5 | 收尾阶段 |

**分阶段路线**
- **Phase 0（文档与规范）**：GUIDE + DEVELOPMENT（进行中）。
- **Phase 1（MVP）**：T1 → T2 → T3 → T4 → T5，跑通「导入 .md → 提交（带语义版本）→ 选两版看句子级 diff」。
- **Phase 2（增幅）**：并排双栏、统计摘要、时间线浏览。
- **Phase 3（自动化）**：监听文件自动检测变更（可选自动提交）。
- **Phase 4（形态演进）**：封装为 Obsidian 插件，复用存储与引擎。
- **Phase 5（收尾优化）**：T7 前端优化，集中打磨样式 / 交互 / 响应式 / 无障碍等体验细节，留到功能就绪后统一处理，避免过早分散精力。

> 用户原则：MVP 先把最小技术栈跑通，再按需增加技术栈强化功能。任务权重用于决定实现优先级，不僵化排期。
